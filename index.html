Introduction

The Printix cloud print api is intended for use by applications that wish to push printjobs, in either PDF or a native Print format(eg, PCl) into a Printix queue.

Creating and printing a document is a multistage process, consisting of:

  1. Fetching a list of printers (See List Printers)

  2. Selecting a printer (See Printer info)

  3. Creating a job for that printer (See Submit a job)

  4. using the links returned by in the response body during step 3 to perform the upload (See Complete Upload)

  5. Inform the Printix api that the document has been uploaded and is ready to be printed.

Each of these four steps are detailed in their respective sections of this document.

Api Conventions

This api borrows some resource models from the, now deprecated, Google cloud print rest api GCP Application end-points. However due to inherent differences in the underlying functionality it does not function as a drop-in replacement for the Google Cloud Print service.
Terminology: The word "Resource" is used to refer to a single entity (printer, job etc) in it’s raw form. The word "Document" is used to describe a complete response, so a resource or multiple resources and whatever container elements they may be returned in.

This api makes use of the HAL document format As such, it is intended to be discoverable by only having knowledge of the API entrypoint (see root request). All successful responses contains links that allow exploring the rest of the Api. As such, consumers of the api are encouraged to not try to manually construct urls, but instead make use of the ones returned by previous api responses. An example response looks like:

{
  "_links" : {
    "self" : {
      "href" : [the url of the returned document] ,
      "templated" : true
    }
  },
  ...object properties in json
}

Resource lists (jobs, printers) are returned in a paginated container format, with information about the current page(page number, page size, total elements and total page count) and links to next and previous page if these exist.

{
  "_links" : {
    "self" : {
      "href" : [the url of the returned document],
      "templated" : true
    },
    "next" : {
       "href" : [link  to next page of element],
    }
  },
  "success" : true,
  "message" : "OK",
  "printers" : [ ],
  "page" : {
    "size" : 100,
    "totalElements" : 101,
    "totalPages" : 2,
    "number" : 1
  }
}

See the individual resource descriptions below for a more detailed view of resource structures.

Authentication

The service interacting with Printix must first aquire a set of service tokens. This is done using the Client Credentials flow of oauth 2

The service is provided, out of band, with a credential set consisting of a clientid and a client secret. These are posted to the printix token endpoint.

$ curl 'https://auth.printix.net/oauth/token'  -X POST \
    -H 'Content-Type: application/x-www-form-urlencoded'\
    -d 'grant_type=client_credentials&client_id={clientId}&client_secret={clientSecret}

In case of success the response will be

Content-Type: application/json;charset=UTF-8

{
  "access_token": "5a8c4ec4-ff70-4f4c-b088-29a8cae38062",
  "refresh_token": "e6f05338-9d27-466b-afa8-0ce72945da7d",
  "expires_in": 599
}

Submit a job

The /submit end-point is used to initiate the upload of a document for printing.

Instead of taking document data to be uploaded and sent to the printer, this end-point takes in a title and a ticket describing the print job. It returns the job’s metadata, as well as generated links to upload the job to cloud store via a call to the /completeUpload end-point.

Request parameters:

Parameter Description

title

The title of the printjob

content

A ticket, in CJT format, describing the job

Response:

This API call returns a boolean success indicator, a description of the document with metadata, as well as a uploadCompleted link to be used with the /completeUpload end-point.

Below is a breakdown of the fields returned in the response.

Path Type Description

job.id

String

The id of the generated printjob

job.createTime

Number

Unix timestamp for when the job was created

job.printerId

String

The Id of the printer the job was sent to

job.updateTime

Number

Unix timestamp of when this job was last updated(for new jobs, this will be the same as createTime

job.status

String

Current state of the job

job.ownerId

String

The id of the owner of this job. For jobs created with client credentials this will always be the client_id

job.contentType

String

The content type of the uploaded document(currently always PDF or null

job.title

String

The title of the printjob. matching the request parameter given when creating the job

_links.uploadCompleted

Object

Link to be used once a job has completed uploading, to initiate the print process

uploadLinks

Array

document upload link

uploadLinks.[0].url

String

The actual url to upload the document data (Using http PUT)

uploadLinks.[0].headers

Object

Any additional headers to add to the PUT request

Retrieve Jobs

The /jobs end-point is used to retreive a list of jobs for the authenticated user. There are two ways to call the /jobs end-point, as a request can be made with or without specifying a queue ID.

In the first case, a list of all print jobs for a given user is returned; in the second, the call returns a list of jobs for the printer that was specified instead.

Request parameters

This request can take the following optional parameters:

Parameter Description

page

The integer value of the page of result we want to get, defaults to 0 if not set.

pageSize

An integer value used to limit how many results are shown on a page.

This end-point returns a boolean success indicator and a paged list of jobs.

Retrieving and deleting a single job

The /job end-point is used to retrieve a single job, while /delete can be used to remove a job from a printer. Both end-points take no parameters.

/job returns a paged singleton list containing a single job, while /delete returns a boolean success indicator.

Printer info

The /printer end-point (singular) can be used to receive all fields of a specified printer, including its capabilities. It takes no request parameters, and returns boolean success indicator and a paged singleton list containing the requested printer.

As a printer resource contains a lot of different fields whose meaning could be unclear, the table below is provided for the user’s convenience:

List

printer.capabilities.supported_content_type

List

A list of content types/print formats supported by the printer.

printer.capabiliities.copies

Object

Describes the default and maximum number of copies supported by a printer.

printer.capabilities.color

List

A list of color options supported by the printer.

printer.capabilities.vendor_capabilities

List

A list of objects describing whether the printer supports releasing print jobs from the cloud, and various localization options.

_links

Object

A json object containing API links for both the printer that was returned, as well as a /submit link.

printer.jobs

Object

A json object containing a list of jobs associated with the returned printer.

Complete Upload

This end-point informs Printix that the job upload is complete. If the job was created with the "printLater" vendor capability set to false(which is the default if no value was specified), this will also cause the printjob to be sent to the printer. It uploads the job to cloud storage and queues the job for printing via the Printix Cloud.

This end-point takes no request parameters; It returns a boolean success indicator, alongside metadata for the job indicating that it has finished uploading and is being processed.
